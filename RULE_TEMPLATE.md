# 规则标题

## 适用范围

- 说明这条规则适用于什么代码、什么任务、什么目录。
- 说明这条规则不适用于什么场景，避免误伤。
- 如果是前后端混合项目，写清楚只约束前端还是后端。

## 强制要求

- 用明确语气写“必须 / 禁止 / 仅允许”。
- 不要只写抽象原则，要写成可执行要求。
- 一条 bullet 只表达一个约束点。

## 边界说明

- 说明允许的例外情况。
- 说明在什么条件下可以不按默认路径处理。
- 如果没有例外，也可以省略这一节。

## 违规示例

```ts
// ❌ 错误写法 1

// ❌ 错误写法 2
```

## 正确示例

```ts
// ✅ 正确写法 1

// ✅ 正确写法 2
```

## 输出前自检

- 当前输出是否违反了这条规则？
- 有没有出现“看起来差不多，但本质仍违规”的变种写法？
- 是否已经按正确结构落地，而不只是口头说明会这么做？

## 编写建议

- 优先写高频问题，不写低频口号。
- 优先写可检查规则，不写纯主观审美判断。
- 如果这条内容只在特定任务触发，应该改做 `skill`，而不是 `rule`。

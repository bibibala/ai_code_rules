# Debug Workflow

## 何时使用

在以下场景使用本技能：

- 用户反馈“报错了 / 不生效 / 和预期不一致”
- 需要定位接口、状态、渲染、样式、权限、路由等问题
- 需要分析回归问题或修复后做最小验证
- 需要先找根因，再决定是修代码、补保护还是调整数据

适用边界：

- 仅适用于前端报错、前端异常行为、前端页面或交互问题排查。
- 不适用于后端服务报错、数据库异常、消息队列、部署链路等非前端故障。

不适用于：

- 明确需求新增且不存在异常行为的问题
- 纯样式美化且没有故障排查需求的任务

## 工作原则

### 1. 先复现，再下结论

- 优先收集可复现现象、报错信息、触发条件、影响范围。
- 没有证据前，不要直接假设根因。
- 如无法完整复现，也要记录“已知现象”和“未确认点”。

### 2. 先定位层级，再缩小范围

按以下顺序排查：

1. 输入是否正确：参数、路由、权限、环境变量、用户操作
2. 数据是否正确：接口返回、状态存储、转换逻辑
3. 渲染是否正确：条件分支、组件生命周期、样式覆盖
4. 边界是否缺失：空值、失败态、竞态、重复触发

### 3. 优先最小修复

- 优先修根因，不优先堆补丁。
- 修复范围只覆盖受影响链路，避免顺手做无关重构。
- 若必须加保护分支，说明它是在防什么问题。

### 4. 修完必须验证

- 至少验证：问题已复现的主路径不再出错。
- 同时检查最容易受影响的一条相邻路径，避免修一处坏一处。
- 若无法完整验证，明确说明剩余风险。

## 推荐输出结构

1. 现象：用户看到什么、何时出现
2. 定位：问题发生在输入、数据、渲染还是边界
3. 根因：哪一行逻辑或哪类约束导致问题
4. 修复：本次改了什么，为什么这样改
5. 验证：怎么证明修好了，剩余风险是什么

# 复杂实现工作流（Karpathy 风格）

## 定位

- 本 skill 用于“前端复杂任务怎么推进”，不是默认常驻的全局编码规则。
- 通用编码原则（如代码优先、保持简单、可读性优先）应放在 `rules/ai-coding-principles.md` 中长期生效。
- 当任务需要分阶段落地、快速验证、强调可观测性与交付质量时，再启用本 skill。

## 适用范围

- 仅适用于前端功能开发、前端页面改造、前端交互链路重构等前端任务。
- 不适用于后端服务设计、数据库建模、服务间调用链治理等非前端任务。

## 何时使用

- 新功能从 0 到 1，且链路较长
- 复杂页面、复杂交互、复杂状态流改造
- 多模块联动或需要明确分阶段交付
- 需要先打通主链路，再逐步补齐边界、体验和性能
- 需要高质量输出，并希望明确验证方式

## 不适用场景

- 简单 bug 修复
- 小范围文案、样式、配置修改
- 纯机械性替换
- 明显无需分阶段设计的小任务

## 工作方式

### 1. 先给最小可运行版本

- 第一阶段先打通主链路，确保“能跑、能看、能验证”。
- 先解决最关键路径，不在一开始补齐所有边界。
- 第一版代码应尽量短、直观、容易改。

### 2. 按阶段增强

复杂任务按以下顺序推进：

1. 最小版本：主链路跑通
2. 加强版本：补齐边界、错误处理、空态/异常态
3. 完整版本：补齐体验、性能、可维护性

### 3. 强调可观测与可验证

- 不做黑盒实现，关键流程应能被清楚追踪和验证。
- 优先通过清晰数据流和明确返回结构提升可读性。
- 如项目需要观测能力，使用统一 `logger` 或埋点封装，不直接写 `console.*`。

### 4. 快速反馈

- 每个阶段都应附带最小验证方式。
- 优先提供测试思路、复现步骤或最小输入输出样例。
- 修改完成后，应能快速证明主链路可用。

### 5. 控制复杂度

- 如果方案已经明显过重，应及时回退到更简单的实现。
- 不为了“架构完整”牺牲当前可交付性。
- 只有在复杂度真实出现时，才继续拆层和抽象。

## 推荐输出结构

1. 问题分析：简短说明主链路、风险点、边界
2. 分阶段方案：先做什么，再补什么
3. 代码实现：优先给最小可运行版本
4. 验证方式：测试、步骤或最小样例
5. 后续增强：体验、性能、可维护性优化项

## 与本仓库其他内容的关系

- 默认工程约束看 `rules/`
- UI 场景联动 `skills/ui-design`
- 需求分析、设计、任务拆解联动 `skills/spec-workflow`

## 与前端规则对齐

- API 获取：禁止循环请求拼分页/补齐全量；需要批量由后端提供批量接口。
- 选项数据：下拉/选项类组件必须用 list/专用 options 接口，不允许前端拼分页。
- 错误处理：业务层不写 `try/catch`，失败用“可判定结果”显式处理。
- 包管理：统一使用 `pnpm`；复杂功能优先评估成熟 npm 包后再决定是否自研。

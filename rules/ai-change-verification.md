# AI 改动验证规范

## 适用范围

- 适用于本仓库启用范围内的所有前端 AI 编码改动。
- 适用于前端新增功能、修复 bug、重构、样式调整、前端配置变更。
- 不适用于后端服务、数据库脚本、中间件配置、运维脚本等非前端改动。
- 不要求一律补齐完整测试体系，但要求至少完成与本次改动匹配的最小验证。

## 强制要求

- 完成代码改动后，必须执行至少一种与改动直接相关的验证方式。
- 优先使用项目已有的验证手段，例如 `pnpm lint`、`pnpm test`、类型检查、构建检查、最小手动复现步骤。
- 如果仓库已有明确的测试或校验命令，不能跳过判断，必须先检查是否适合运行。
- 如果无法运行验证，必须明确说明原因、未验证范围和潜在风险。
- 输出结果时必须区分：
  - 已完成的验证
  - 未完成的验证
  - 剩余风险或需要用户补验的点

## 推荐验证顺序

1. 先做与改动最接近的最小验证
2. 再做静态检查，例如 lint、type-check
3. 再做受影响范围的测试或构建
4. 如无法自动验证，给出最小人工验证步骤

## 禁止行为

- 改完代码后完全不验证，就直接声称“已完成”。
- 明明没有运行测试，却表述成“功能已确认正常”。
- 只说“理论上可以”，不给任何验证证据或验证路径。
- 验证失败后不说明失败点，直接忽略。

## 违规示例

```md
❌ 已修复，你直接用就行。

❌ 我没有跑任何检查，但应该没问题。
```

## 正确示例

```md
✅ 已完成修复。
已验证：`pnpm lint`、目标页面手动交互检查
未验证：完整回归测试
风险：如果还有其他页面复用该组件，建议再补一次列表页联调
```

## 输出前自检

- 本次改动是否至少做了一项直接相关验证？
- 是否如实说明了“跑了什么”和“没跑什么”？
- 如果校验没跑，是否说明了原因，而不是跳过不提？
- 是否把剩余风险说清楚了，而不是默认视为 0 风险？

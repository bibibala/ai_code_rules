# 前端依赖策略规范

## 适用范围

- 仅适用于前端工程（Web / Electron / H5 / 小程序前端）。
- 不适用于后端服务、数据库脚本、中间件部署脚本。

## 包管理器统一规范

- 所有前端项目统一使用 `pnpm`。
- 禁止使用 `npm` 或 `yarn` 执行安装、卸载、运行等包管理操作。
- 常用命令统一为：
  - 安装依赖：`pnpm add <pkg>`
  - 安装开发依赖：`pnpm add -D <pkg>`
  - 移除依赖：`pnpm remove <pkg>`
  - 安装项目依赖：`pnpm install`
  - 执行临时脚手架：`pnpm dlx <tool>`

## 复杂功能实现原则

- 当功能复杂度较高时，禁止直接手搓实现，必须先评估是否可引入成熟 npm 包。
- 优先“评估复用”，再“自研实现”。
- 以下场景默认先查包：
  - 富文本编辑器、图表、拖拽排序、虚拟列表
  - 日期时间处理、表单校验、文件上传与预览
  - 国际化、状态机、数据可视化、动效库

## 决策要求

- 当需求落在复杂能力范围内时，必须先判断“是否已有成熟 npm 包可用”，再决定是否自研。
- 如果选择引入依赖，必须说明“为什么选它，而不是手搓”。
- 如果选择自研，必须说明“为什么不选现有包”。
- 不允许跳过评估，直接凭直觉手写复杂基础能力。

## 依赖引入评估标准

- 活跃度：是否持续维护、最近更新时间是否合理。
- 可靠性：下载量、社区使用度、issue/PR 响应情况。
- 合规性：license 是否满足项目要求。
- 兼容性：是否兼容当前框架与构建工具（Vue / Vite 等）。
- 成本：体积、按需加载能力、学习与二次封装成本。
- 安全性：是否存在已知高危漏洞。

## 落地要求

- 引入新依赖时，需要在说明中给出“为什么不用手搓”的简要理由。
- 若最终选择自研，也需要说明“为什么不选现有包”。

## 违规示例

```md
❌ 需求：做拖拽排序
处理：直接自己写一套拖拽底层逻辑
问题：跳过了成熟库评估
```

## 正确示例

```md
✅ 需求：做表单校验
处理：先评估成熟库，再决定是否引入
说明：当前场景规则复杂、动态联动多，使用成熟库能显著降低边界处理成本
```

## 输出前自检

- 当前需求是否属于富文本、图表、拖拽、虚拟列表、校验、国际化等复杂能力？
- 是否已经先做“评估复用，再决定自研”？
- 如果引入新依赖，是否说明了引入理由？
- 如果坚持自研，是否说明了不选现有包的原因？
- 是否全程使用 `pnpm`，没有出现 `npm` 或 `yarn`？

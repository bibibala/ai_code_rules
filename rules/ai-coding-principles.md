# AI 编码强制原则（简洁、优雅、可读）

## 适用范围

- 仅适用于本仓库启用范围内的前端 AI 编码任务。
- 作为前端通用实现原则使用，可与前端专项规则同时生效。
- 不适用于后端服务、数据库脚本、中间件、运维脚本或非前端研发任务。
- 不替代具体框架、目录结构、接口约束等专项规则。

## 强制要求

### 代码优先

- 优先产出可运行、可验证的实现，而不是只给思路。
- 示例必须完整、可复制，至少应贴近真实运行上下文。
- 结论应体现在代码和可验证行为上。

### 保持简单

- 必须优先最简单可行实现，避免过度设计。
- 禁止为了“看起来高级”而增加不必要抽象。
- 如果封装不能明显降低复杂度或复用成本，就不要封装。

### 禁止臃肿实现

- 禁止为了“防止出错”预先堆叠多层包装、适配器、兜底分支、状态中转或重复判空。
- 禁止同一层级里出现“先包一层对象、再拆一层对象、再转一层对象”的无意义数据搬运。
- 禁止简单逻辑拆成过多小函数、过多 hooks、过多 computed/derived 层，导致阅读需要来回跳转。
- 必须直接表达业务意图，不要写“看起来很安全、实际很啰嗦”的代码。
- 防御性代码必须有真实触发场景；没有明确风险证据时，不要先加一整套保护壳。

### 可读性优先

- 单个函数保持单一职责。
- 命名必须表达语义，避免技巧化写法。
- 嵌套超过 3 层时，优先改为早返回或拆分函数。

### 命名简洁

- 命名必须简短、直接、可理解，优先使用项目里已经稳定存在的短语。
- 禁止使用过长命名来硬塞实现细节、时序信息或防御意图。
- 能用 `list`、`detail`、`fetchUser`、`isLoading` 说清的，不要写成超长复合名。
- 不要为了“绝对精确”把一个变量名写成一句话。

### 数据优先

- 必须优先围绕数据结构设计，而不是堆叠流程分支。
- 优先使用映射表、配置驱动等清晰结构替代大段 `if/else`。

### 避免隐式行为

- 禁止依赖隐式副作用、魔法默认值、难以推断的技巧代码。
- 必须优先使用清晰 API 和显式流程表达逻辑。

### 可验证

- 每段核心代码都必须能快速验证。
- 优先写最小可复现用例、小输入、小输出的验证方式。
- 关键链路应具备可追踪、可定位、可检查的特征。

### 性能意识

- 默认实现应具备基本性能意识。
- 避免重复计算、无意义的大对象复制和明显可规避的额外渲染。
- 仅在热点路径或证据充分时增加缓存、剪枝等优化。

## 禁止行为

- 只给概念说明，不给落地代码。
- 输出不可执行的伪代码作为最终结果。
- 用异常做日常业务流程控制。
- 为简单问题引入多层抽象和复杂架构。
- 为了降低心理风险而堆叠多层无必要兜底代码。
- 用超长命名、重复包装、重复转换制造“看起来很严谨”的复杂度。

## 违规示例

```ts
// ❌ 无意义包装过多
function createSafeUserViewModelWrapper(response) {
  const safeResponseData = response?.data ?? {}
  const normalizedUserData = normalizeUserDataSafely(safeResponseData)
  return {
    userViewModel: {
      displayUserName: normalizedUserData?.userDisplayName ?? '',
    },
  }
}

// ❌ 命名过长
const shouldHandleUserListRequestLoadingStateInCurrentPage = true
```

## 正确示例

```ts
// ✅ 直接表达意图
function getUserName(data) {
  return data?.name ?? ''
}

// ✅ 简洁命名
const isLoading = true
const userList = []
```

## 输出前自检

- 这次输出是否给了可运行或接近可运行的实现，而不是只给思路？
- 方案是否足够简单，还是为了“高级感”加了不必要抽象？
- 是否加入了没有明确必要的包装层、转换层、兜底层？
- 函数职责、命名和控制流是否足够直观？
- 命名是否已经足够短，还是长到像一句解释？
- 是否存在隐式副作用、魔法值或难以推断的技巧代码？
- 是否附带了最小验证方式，能快速证明改动有效？
